# Writeflow 产品方案

## 产品定位

Writeflow 是一个本地优先、human-in-the-loop 的长文写作工作台，当前从小说续写场景切入。产品通过多个专业写作 Agent 协作，同时把关键创作决策保留给人类。

Writeflow 的核心承诺不是“一键生成”，而是提供一个可控的写作流水线：项目记忆、上文上下文、Agent 产物和人工审批闸门共同协作，生成更一致、更可控的长文草稿。

## 目标用户

- 需要跨章节保持连续性的长篇小说作者。
- 希望使用 AI 辅助，但仍要亲自审批方向、大纲、正文和记忆更新的创作者。
- 正在探索结构化 AI 写作流程的编辑或写作团队。
- 未来可扩展到故事编辑、提案写作等相邻场景的用户。

## 产品原则

- 本地优先存储：项目数据默认保存在用户机器上。
- 人工审批闸门：重要创作决策必须由用户显式选择。
- 上文硬依赖：续写必须基于 `manuscript/context.md`，避免脱离原文另起炉灶。
- 可迁移记忆：长文记忆使用 Markdown，而不是不可读的数据库 Blob。
- Agent 专业分工：每个 Agent 只负责明确的写作职责，并产出类型化 artifact。
- 连续性优先：生成必须尊重上文、canon、时间线、文风和开放线索。
- 保守记忆更新：Agent 只提出记忆补丁，由用户批准后再应用。
- 可见过程：生成时展示当前 Agent、执行阶段和耗时，降低黑盒感。

## 当前主工作流

当前主工作流是 `novel_continue`：基于已有上文和项目记忆续写小说。

1. 用户创建小说项目。
2. Writeflow 初始化记忆模板和项目存储目录。
3. 用户填写或编辑项目记忆。
4. 用户在“续写上文”区域粘贴并保存最新上文。
5. Muse 读取上文与记忆，生成多个续写方向。
6. 闸门 1：用户选择故事方向。
7. Architect 把所选方向转成章节 beat sheet。
8. 用户阅读大纲；需要时点击“编辑大纲”进入编辑模式直接修改。
9. 闸门 2：用户确认大纲后进入正文生成。
10. Scribe 按大纲场景分段生成正文草稿变体。
11. 用户可按场景查看并调整续写部分（默认只读，点按钮进入编辑）。
12. Editor 按场景润色草稿变体。
13. Critic 按变体和文本块审查故事、canon、时间线、人物和文风问题。
14. （可选）修订循环：用户在审稿面板勾选要修复的问题，交编辑智能体修订，生成新的润色版本并可再次审稿。
15. 闸门 3：用户选择终稿变体，累计进入全文。
16. （可选）用户手动编辑该章正文/标题，或在编辑态里圈选一段交 AI 重写（原位替换）；改后自动重算该章概要并重同步其记忆。也可删除某章（带二次确认）。
17. Archivist 基于选定终稿的分块摘要提出记忆更新补丁。
18. 用户批准并应用记忆补丁。
19. 用户点「继续写下一章」回到构思，自动承接「原始上文 + 已选章节全文」继续下一章。

每一章的方向 / 大纲 / 草稿 / 润色 / 审稿 / 终稿都按章节归档，可在「章节档案」回看。

## 自动续写模式（Autopilot）

主工作流逐闸门把关，适合需要逐步审稿控场的作者。Autopilot 是叠加在同一套 Agent 与 artifact 之上的全自动路径，面向「先一次性产出多章草稿、再回看打磨」的场景。

输入：本程聚焦目标（可留空）+ 可选的开头几章要点。章数不再由用户手填，而是由路线图规划器根据创作纲领、续写上文、现有记忆和人物/伏笔/结局基调自行判断。

执行：

1. Roadmap Planner 先生成从当前进度到结局的**续写路线图**，并作为可见 `chapter_plan` 产物展示；用户审阅路线图后再点「基于此规划开写」。
2. 逐章自动走完构思→大纲→写作→润色→审稿→终稿→归档全流程；各闸门由模型自动决策（方向自动采纳 Muse 推荐，终稿自动取审稿最强变体）。
3. 质量兜底：每章 写→润→审，verdict 非 pass 时自动按 issues 修订重审，默认最多 2 轮。修订会按每条 issue 的建议采用不同动作（删减、弱化、替换、澄清设定等），复审会看到上一轮修订记录以避免同类主观意见反复漂移；轮次耗尽后若只剩 minor 可继续，残留 blocker/major 才停在该章并报告。
4. 分批软闸门：按路线图写作时每完成 5 章会正常停下待审；用户可手动继续下一批，若页面保持打开且 20 分钟无操作则自动继续。软闸门不是失败，不消耗自动重试次数。
5. 章间记忆：多章连跑期间**自动应用**记忆补丁，让后章承接前章新设定。
6. 中途卡住或被中止：停在出问题的那章，**已成章全部保留**，可转回手动流程继续。

已有路线图时，可直接「基于此规划开写」，跳过重新规划、照现有路线图逐章执行。

**断点续跑**：任务失败/中断后可从断点继续，不必整批重来。批次进度（路线图、各章已完成阶段、审稿修订进度、自动重试计数）落盘到 `autopilot/current.json`；续跑跳过已定稿章、复用失败章内已完成阶段（方向/大纲/草稿/润色/审稿修订中间态），只补没跑完的部分。触发方式：面板「继续」按钮（手动）；意外失败（模型错/超时/进程中断）约 1 分钟后自动重试，最多 3 次（计数持久化）；`quality`/`duplicate` 是有意停下报告，不自动重试。软闸门 `gated` 可手动或 20 分钟超时自动续，不计入失败重试。

边界：路线图章数有安全上限（防模型规划离谱长篇），但不再是固定“最多 10 章”；单批仍按 5 章停顿以控制成本与可审阅性。自动应用记忆带来 canon 被自动改写的风险（已在「连贯优先」前提下接受，保留 applied 记录可追溯）。

## 人工闸门

### 闸门 1：方向选择

用户选择哪个 Muse 方向进入下一章规划。这样系统不会在关键故事走向上静默替用户做决定。

### 闸门 2：大纲确认

用户在生成正文前确认或编辑 Architect beat sheet，保证章节因果结构和场景逻辑仍由人控制。

### 闸门 3：终稿选择

用户选择最终的编辑后正文变体。被选中的 artifact 会成为后续记忆更新的依据。

### 记忆补丁审批

Archivist 在终稿选定后提出项目记忆变更。应用补丁是独立审批步骤，避免 canon 被自动改写。

## Agent 角色

### Roadmap Planner（续写路线图）

自动续写的故事级路线图 Agent。

职责：

- 根据创作纲领、续写上文、已成章记忆和用户给定的开头要点，自行判断从当前进度到结局大约需要多少章。
- 生成可审阅的 `chapter_plan` 路线图产物，每章给出标题、要点和聚焦点。
- 用户给了开头几章要点必须采纳，其余章节按人物弧线、伏笔规划和结局基调合理编排。
- 尊重续写上文与项目记忆，不重复已成章内容。
- 不写正文，仅作为可见路线图驱动后续逐章流程。

### Muse

续写方向构思 Agent。

职责：

- 基于续写上文和项目记忆生成多个故事方向选项。
- 尊重 canon、时间线、当前状态、文风笔记和开放线索。
- 不写正文。
- 标注受影响线索、人物压力、风险和下一拍。

### Architect

章节结构规划 Agent。

职责：

- 把一个已批准的 Muse 方向转成章节 beat sheet。
- 定义每个场景的目的、冲突、情绪转折、信息释放、线索推进和结尾钩子。
- 暴露连续性检查项和结构风险。

### Scribe

正文草稿 Agent。

职责：

- 把已批准的 beat sheet 转成正文。
- 按场景分段生成正文，保持与大纲对应。
- 生成两个真正有区别的草稿变体。
- 保留已批准的故事逻辑。
- 避免泛化 AI 腔和空洞抒情。

### Editor

修订润色 Agent。

职责：

- 按场景润色正文，降低长文本请求失败风险。
- 改善节奏、清晰度、转场、对话和文风一致性。
- 保留每个变体的不同策略。
- 不引入新的情节转折或 canon 事实。

### Critic

对抗式审稿 Agent。

职责：

- 按变体和文本块审查长文内容。
- 查找 canon 矛盾、时间线错误、人物崩坏、情绪无支撑、线索推进不足、因果薄弱和泛化文风。
- 使用 blocker、major、minor 分级报告问题并提供证据。
- 建议当前稿件应该通过、修订还是拒绝。

### Archivist

记忆维护 Agent。

职责：

- 在终稿选定后先把终稿分块摘要成记忆相关事实。
- 基于摘要提出保守的记忆更新。
- 在终稿文本支持的前提下更新进度状态和开放线索。
- 标记变更需要人工审批。
- 不直接改写正文，也不自动修改 canon。

## 记忆模型

小说项目会初始化 Markdown 记忆文件：

- `memory/plan/blueprint.md`（创作纲领，写正文前确定的前瞻记忆）
- `memory/canon/characters/_template.md`
- `memory/canon/world.md`
- `memory/canon/timeline.md`
- `memory/progress/state.md`
- `memory/progress/open_threads.md`
- `memory/style/voice.md`
- `memory/style/taboo.md`
- `manuscript/context.md`

产品把记忆视为作者可编辑的源材料。Agent recall 会读取这些文件，并把它们以摘录形式注入提示词上下文，避免长项目记忆导致模型请求过大。记忆页按 Markdown 渲染阅读、点「编辑」进入文本框。

**章节可追溯记忆**：timeline、open_threads、world、characters 等累积型记忆中，每一章贡献的内容用标记块（HTML 注释分隔，阅读时不可见）按章存储。这样编辑或重写某章后能精确替换该章对应的记忆，而不是无限追加、堆叠重复；`state/voice/taboo` 属"最新视角"的全局文件，改中间章不回退。前情（开篇上文）会在首次归档时自动写入记忆，让 timeline 从故事第一句起就完整。若历史记忆已重复或错乱，可「从章节重建记忆」一键修复（清空机器写入块，保留手写基底，逐章重建）。

## 当前 UX 形态

工作台是三栏写作 IDE：

- 左栏：工作流阶段管线（构思→大纲→写作→润色→审稿→终稿→归档），按已有产物派生真实状态——已完成 / 当前步 / 锁定，点击可切换中栏聚焦的阶段。
- 中栏：浅色文档区，只展示当前阶段的面板与操作；每个阶段默认展示最新 artifact，长正文支持折叠/展开（展开也有最大高度、内部滚动）。终稿阶段的章节列表点选后只显示该章正文，并可手动编辑/圈选重写/删除。
- 右栏：续写上文编辑、记忆文件快捷入口、已写全文（前情 + 各章）、生成过程面板。

生成过程面板通过 SSE 实时展示当前智能体、子步骤、模型逐字输出与耗时，刷新页面不丢失。另有「章节档案」页按章回看每一章的完整创作过程。

## 当前产品缺口

- 工作流编排仍是面板驱动 + 按产物派生状态，尚未落成单一、有状态的 run 状态机。
- 数据库中已有 gate 表，但闸门仍以「按产物存在性派生」呈现，未作为一等记录持久化驱动。
- schema 中已有 `novel` 之外的项目类型，但还没有完整产品流程。
- 设置页和 provider 配置还需要成为面向用户的模型设置流程。
- 手动流程的修订循环仍为手动触发、手动重审；自动迭代到收敛目前只在 Autopilot 路径内实现（有最大轮数与“只剩 minor 可放行”的收敛判定）。
- 旧数据（早于父链/章节化的产物）在章节档案里只能看到终稿，上游阶段缺失。

## 建议的下一步产品迭代

1. 把面板序列升级成可见、可持久化的 workflow run 状态机，展示当前步骤、依赖和下一步动作。
2. 把 gate 决策持久化为一等记录，并用 active run 派生 UI 状态。
3. 把 Autopilot 的修订收敛机制扩展到手动审稿流程（手动触发但可复用同样的复审收敛锚）。
4. 提升记忆可见性，展示哪些文件会进入 recall，以及哪些记忆缺失会影响输出。
5. 把 provider 配置做成面向用户的模型设置流程。
6. 在小说续写闭环稳定后，再扩展故事编辑或提案写作工作流。
